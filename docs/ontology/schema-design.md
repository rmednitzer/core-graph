# Schema design: six-layer unified ontology

## Introduction

core-graph consolidates six distinct knowledge domains into a single PostgreSQL
graph powered by Apache AGE (openCypher) and pgvector (HNSW embeddings). Rather
than maintaining separate databases per domain, a unified graph lets analysts
traverse from a STIX threat actor through the security events it generated,
into the OSINT articles that first reported it, across the compliance controls
it violates, through any AI-assisted investigation sessions, and along the
forensic timeline that reconstructs the full incident narrative.

All six layers share one set of bitemporal columns, one entity-resolution
pattern, one Row-Level Security policy hierarchy (TLP-based), and one
authorization stack (Cerbos ABAC + SpiceDB ReBAC). This means a single query
can cross layer boundaries without federation, and every fact in the graph
carries identical provenance and temporal metadata regardless of its origin.

The graph lives in a dedicated AGE graph namespace (`core`). Each layer is
distinguished by vertex and edge labels, not by separate schemas or databases.
pgvector embeddings are stored on vertex properties for semantic search and are
indexed with HNSW for approximate nearest-neighbour retrieval.

## Layer 1: Threat intelligence (STIX 2.1)

Threat intelligence is the primary ingest layer. It models adversary behaviour,
campaigns, indicators of compromise, and defensive tooling using the STIX 2.1
data model as the canonical vocabulary. Sources include OpenCTI, MISP, and
direct TAXII feeds, all arriving through NATS JetStream ingest adapters.

### Vertex labels

| Label | Purpose |
|---|---|
| `ThreatActor` | Named adversary group or individual |
| `Campaign` | Coordinated set of adversary activities |
| `AttackPattern` | Technique or sub-technique (maps to MITRE ATT&CK) |
| `Indicator` | Observable pattern indicating malicious activity |
| `Malware` | Malicious code instance or family |
| `Vulnerability` | Software weakness, typically carrying a CVE identifier |
| `Tool` | Legitimate or dual-use software employed by adversaries |

### Edge labels

| Label | Typical source | Typical target | Meaning |
|---|---|---|---|
| `uses` | ThreatActor, Campaign | AttackPattern, Malware, Tool | Actor employs technique or tool |
| `targets` | ThreatActor, Campaign | Vulnerability, CanonicalOrganization | Adversary targets a weakness or entity |
| `attributed_to` | Campaign, Malware | ThreatActor | Attribution relationship |
| `indicates` | Indicator | ThreatActor, Campaign, Malware | Observable suggests presence of threat |
| `mitigates` | Tool, AttackPattern | Vulnerability | Defensive relationship |

### Retention

5 years from `t_recorded`. After expiry, facts are marked `t_invalid` but
remain in the graph for historical queries.

### Reference

See [stix-mapping.md](stix-mapping.md) for the complete property-level mapping
from STIX 2.1 SDOs, SROs, and SCOs to graph elements.

## Layer 2: Security events (OCSF)

Security events arrive from satellite systems (Wazuh, OpenSearch, EDR agents)
and are normalised through the Open Cybersecurity Schema Framework before
ingestion. This layer captures what happened, when, and where, at machine
speed. High-volume events are filtered and deduplicated at the ingest adapter
before entering the graph; only events that meet a minimum severity threshold
or match an active indicator are promoted to vertices.

### Vertex labels

| Label | Purpose |
|---|---|
| `SecurityEvent` | General normalised event (catch-all for uncategorised OCSF classes) |
| `NetworkActivity` | Connection, flow, or DNS event |
| `AuthEvent` | Authentication or authorisation event |
| `ProcessEvent` | Process creation, termination, or injection |
| `Finding` | Alert, detection, or analytic match |

### Edge labels

| Label | Typical source | Typical target | Meaning |
|---|---|---|---|
| `triggered_by` | Finding | SecurityEvent, NetworkActivity, ProcessEvent | Alert was generated by this event |
| `observed_on` | SecurityEvent | CanonicalIP, CanonicalDomain | Event involved this network entity |
| `detected_by` | Finding | Source (external detector) | Which system produced the finding |
| `correlated_with` | SecurityEvent | SecurityEvent, Finding | Temporal or causal correlation |

### Retention

1 year from `t_recorded`. High-fidelity findings linked to active
investigations are promoted to the forensic timeline layer before expiry.

### Reference

See [ocsf-normalization.md](ocsf-normalization.md) for the full OCSF category
mapping and field extraction rules.

## Layer 3: OSINT

Open-source intelligence feeds, RSS articles, paste-site monitors, and social
media extractions land here. NER (Named Entity Recognition) pipelines extract
structured entities from unstructured text and link them to canonical nodes.
Each extracted entity carries a confidence score reflecting the NER model's
certainty, and a provenance chain linking back to the source document and
extraction timestamp.

### Vertex labels

| Label | Purpose |
|---|---|
| `Article` | Full-text article or blog post |
| `FeedItem` | Lightweight feed entry (RSS, Atom, API) |
| `Source` | Publisher, feed URL, or platform identity |
| `ExtractedEntity` | NER-extracted entity (person, organisation, IP, domain, hash) |

### Edge labels

| Label | Typical source | Typical target | Meaning |
|---|---|---|---|
| `mentions` | Article, FeedItem | ExtractedEntity, ThreatActor | Text references this entity |
| `extracted_from` | ExtractedEntity | Article, FeedItem | Provenance of the extraction |
| `published_by` | Article, FeedItem | Source | Attribution to publisher |
| `corroborates` | Article | Indicator, Finding | OSINT supports an existing finding |

### Retention

5 years from `t_recorded`.

## Layer 4: Audit and compliance

This layer maps organisational controls to evidence records, enabling
continuous compliance demonstration against frameworks such as ISO 27001,
NIS2, DORA, and SOC 2. Evidence records reference artefacts stored in MinIO
WORM buckets, signed with cosign, and anchored in a Rekor transparency log.
The graph structure allows auditors to traverse from a framework requirement
down to individual evidence artefacts and back up to the controls they satisfy.

### Vertex labels

| Label | Purpose |
|---|---|
| `EvidenceRecord` | Timestamped, signed proof of a control being satisfied |
| `ComplianceControl` | Individual control requirement (e.g., ISO 27001 A.8.2) |
| `Framework` | Regulatory or standards framework |

### Edge labels

| Label | Typical source | Typical target | Meaning |
|---|---|---|---|
| `satisfies` | EvidenceRecord | ComplianceControl | Evidence demonstrates control fulfilment |
| `evidenced_by` | ComplianceControl | EvidenceRecord | Reverse navigation |
| `controls` | Framework | ComplianceControl | Framework defines this control |
| `audited_by` | EvidenceRecord | CanonicalPerson, CanonicalOrganization | Who performed the audit |

### Retention

10 years from `t_recorded`. Evidence records are stored in MinIO WORM buckets
with cosign signatures and recorded in Rekor for tamper-evident verification.

## Layer 5: AI conversation memory (Graphiti-inspired)

AI agents interacting through the MCP server accumulate knowledge across
sessions. Inspired by the Graphiti framework, this layer uses bitemporal
episodic memory so that facts can be superseded without loss of history. The
agent can recall what it knew at any point in time and trace how its
understanding evolved. The `supersedes` edge creates an explicit version chain,
making it possible to reconstruct the agent's belief state at any historical
moment by filtering on `t_valid` and `t_superseded`.

### Vertex labels

| Label | Purpose |
|---|---|
| `Session` | Bounded conversation or agent task |
| `Episode` | Segment within a session (a turn, a tool call, a reasoning step) |
| `ExtractedFact` | Discrete proposition derived from conversation |
| `ConceptEntity` | Abstract concept or named entity referenced across sessions |

### Edge labels

| Label | Typical source | Typical target | Meaning |
|---|---|---|---|
| `derived_from` | ExtractedFact | Episode | Fact was extracted during this episode |
| `supersedes` | ExtractedFact | ExtractedFact | Newer fact replaces older one |
| `valid_during` | ExtractedFact | (temporal property) | Bitemporal validity window |
| `mentioned_in` | ConceptEntity | Episode, Session | Concept appeared in this context |

### Retention

Indefinite. AI memory is never automatically expired. Manual curation through
the MCP server can mark facts as superseded but the original vertices remain
for auditability.

## Layer 6: Forensic timeline

Incident response and digital forensics require a strict causal ordering of
events. This layer reconstructs what happened during an incident by linking
events from other layers into causal chains with explicit temporal ordering.
Timeline events typically reference vertices from Layers 1, 2, and 3 through
cross-layer edges, assembling a coherent narrative from disparate data sources.
Evidence links point to preserved artefacts (disk images, packet captures, log
extracts) stored in MinIO WORM buckets with integrity guarantees.

### Vertex labels

| Label | Purpose |
|---|---|
| `TimelineEvent` | A single event placed on the forensic timeline |
| `CausalChain` | Named sequence of causally related events |
| `EvidenceLink` | Reference to a preserved evidence artefact (disk image, packet capture, log extract) |

### Edge labels

| Label | Typical source | Typical target | Meaning |
|---|---|---|---|
| `caused_by` | TimelineEvent | TimelineEvent | Direct causal relationship |
| `preceded_by` | TimelineEvent | TimelineEvent | Strict temporal ordering without causal claim |
| `contemporaneous_with` | TimelineEvent | TimelineEvent | Events occurred within the same temporal window |

### Retention

10 years minimum from `t_recorded`. Forensic evidence chains must outlast any
plausible litigation or regulatory inquiry window.

## Cross-layer canonical entities

Four canonical vertex labels exist outside any single layer. They act as
anchor points that unify observations from every layer through entity
resolution. Canonical entities are the only vertices that receive `same_as`
edges, which assert identity equivalence discovered through different resolution
paths (e.g., an IP address seen in a STIX indicator and again in a Wazuh alert).

| Label | Purpose | Example |
|---|---|---|
| `CanonicalIP` | Resolved IPv4 or IPv6 address | `198.51.100.23` |
| `CanonicalDomain` | Resolved fully-qualified domain name | `evil.example.com` |
| `CanonicalPerson` | Resolved human identity (pseudonymised) | `analyst-7a3f` |
| `CanonicalOrganization` | Resolved organisation | `ACME Corp` |

### Cross-layer edges

| Label | Meaning |
|---|---|
| `observed_as` | Links a canonical entity to a layer-specific observation, preserving the original source record, timestamp, and confidence |
| `same_as` | Asserts identity equivalence between two canonical entities discovered through different resolution paths |

Every layer-specific vertex that references an IP, domain, person, or
organisation creates an `observed_as` edge to the corresponding canonical
vertex rather than duplicating the entity. This design ensures that a single
query on `CanonicalIP('198.51.100.23')` returns every threat intel indicator,
security event, OSINT mention, and forensic timeline entry involving that
address.

## Bitemporal model

Every vertex and edge in the graph carries four temporal columns:

| Column | Type | Meaning |
|---|---|---|
| `t_valid` | `timestamptz` | When the fact became true in the real world |
| `t_invalid` | `timestamptz` | When the fact ceased to be true (NULL if still valid) |
| `t_recorded` | `timestamptz` | When the system ingested this fact (write-once) |
| `t_superseded` | `timestamptz` | When a newer version of this fact was recorded (NULL if current) |

### Computed range

A computed column `valid_range` is derived as:

```sql
valid_range tstzrange GENERATED ALWAYS AS (
    tstzrange(t_valid, COALESCE(t_invalid, 'infinity'::timestamptz), '[)')
) STORED
```

This enables range-based temporal queries using PostgreSQL's native range
operators (`@>`, `&&`, `-|-`).

### Indexing

A GiST index on `valid_range` supports efficient temporal containment and
overlap queries:

```sql
CREATE INDEX idx_valid_range ON core.vertices USING GiST (valid_range);
```

### Invariants

- Facts are invalidated, never deleted. Setting `t_invalid` retires a fact
  from current queries while preserving it for historical reconstruction.
- Setting `t_superseded` marks a fact as replaced by a newer version. The
  `supersedes` edge in Layer 5 (AI memory) makes this relationship explicit;
  in other layers the supersession chain is implicit through matching canonical
  keys.
- `t_recorded` is write-once and set at ingest time. It is never modified.

## Cross-layer entity resolution

Entity resolution prevents duplicate representations of the same real-world
object across layers. The pattern is deterministic and reproducible.

### Canonical key derivation

Each canonical entity receives a stable identifier computed as:

```
canonical_key = SHA-256( normalise(type) || ':' || normalise(value) )
```

Where `normalise` applies type-specific rules:

- **IP addresses**: expand to full form (IPv6 long form, IPv4 as-is)
- **Domains**: lowercase, strip trailing dot, apply IDNA encoding
- **Persons**: lowercase, strip whitespace, apply configured pseudonymisation
- **Organisations**: lowercase, strip legal suffixes (GmbH, Ltd, Inc)

### MERGE pattern

When an ingest pipeline encounters a resolvable entity, it executes the
following pattern using AGE Cypher:

```cypher
MERGE (c:CanonicalIP {canonical_key: $key})
ON CREATE SET c.value = $value, c.t_valid = $now, c.t_recorded = $now
ON MATCH SET c.last_observed = $now
```

This creates the canonical vertex if absent or updates its `last_observed`
timestamp if it already exists.

### OBSERVED_AS edges

After the MERGE, the source-specific vertex creates a provenance edge:

```cypher
MATCH (src:Indicator {stix_id: $indicator_id})
MATCH (c:CanonicalIP {canonical_key: $key})
CREATE (src)-[:observed_as {
    source_layer: 'threat_intel',
    confidence: $confidence,
    t_recorded: $now
}]->(c)
```

Every `observed_as` edge carries the source layer, confidence score, and
recording timestamp so that downstream consumers can weight and filter
observations by provenance.
